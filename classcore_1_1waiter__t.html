Подробное описание

Блокировка потока в ожидание некого события

По своей сути класс является инкапусляпцией связки mutex + condition_variable

Основные замечения по реализации:

  • В реализаиции boost::condition_variable для Win32 - очень большой оверхед
  • Реализация boost::condition_variable для Win32 (версия 1.46.1, а может быть и позднее) - ведет себя некоректно (что приводит к "падению" программы)

Поэтому в случае Win32 используются напрямую ф-ции WinApi: SetEvent/WaitForSingleObject и т.п.

Заметки
Все методы Thread-Safe

◆ waiter_t()

core::waiter_t::waiter_t ( bool  manual_reset = false)

Конструктор "guarded suspension" сигнализатора

Аргументы
manual_resetПризнак автоматического/ручного перевода объекта в signalling state после ожидания

Если флаг manual_reset равен true, то создаётся экземпляр объекта, который требует обязательного использования метода \reset для перевода объекта в nonsignalled состояние. Если флаг manual_reset равен false (по умолчанию), то создаётся экземпляр объекта с автосбросом состояния в nonsignalled после использования метода \wait.

◆ wait()

bool core::waiter_t::wait ( int  timeout_ms = 0)

Ждать сигнала

Аргументы
timeout_msМаскимальное время ожидания в мс, если timeout_ms == 0, то время ожидания бесконечно
Возвращает
true если объект был разбужен вызовом методом wake, если же был таймаут ожидания, то возвращает false

Ф-ция блокирует выполнение текущего потока в ожидание, пока пользователь не вызовет метод wake.

Предупреждения
Если до вызова wait объект был разбужен (один или несколько раз) вызовом wake, то wait вернет true. Т.е. поведение аналогично WaitForSingleObject, в котором в отличие от condition_variable проверяется "signalling state" прежде чем заснуть.
